# M21 — Metadata Sidecar System

**Mission ID:** M21
**Prerequisites:** M20 complete. Agents can act on the workspace through transactions.
**Output:** The *novel* V3 feature — a per-file metadata sidecar system. For every source file an agent modifies, a companion markdown document is produced at `.ide/meta/<relative-path>.md` capturing plain-English reasoning: why this code exists, what alternatives were considered, what dependencies matter, what references were consulted, a running history of sessions that touched it. Sidecars are hidden by default in the sidebar (toggleable via settings) and machine-generated by a lightweight local summarizer after each completed agent session. The format is Markdown (LLM-friendly, human-readable, grep-able) with a YAML frontmatter for structured fields. A separate `.ide/tasks.md` serves as a project TODO list agents can read and write.
**Estimated scope:** 2 sessions.

## The Situation In Plain English

This is the mission where we bet on something no mainstream IDE does. Every other editor — Cursor, Windsurf, VS Code + Copilot, Zed — treats the AI conversation as ephemeral: you start a chat, the agent reasons for thousands of tokens, it writes code, the conversation ends, everything except the final diff is discarded. Next week, you ask the same agent about the same code, and it has to rebuild all that reasoning from scratch. Tokens lost. Insight lost. Time lost.

V3's bet: if we *capture* the essential reasoning as a sidecar document next to each code file, that reasoning accumulates. A file that's been worked on for six months by various agents and humans acquires a rich, self-describing history. The next agent to touch it doesn't read a raw `.rs` file cold — it reads the sidecar first, gets oriented in hundreds of tokens, and starts work already informed. That compounding is the long-term asset.

The engineering challenge is getting the summarization loop right. After an agent session (a user prompt + agent response + any tool calls + any committed edits), we need to distill it into something concise, accurate, and persistently useful. The expensive frontier model already did the expensive reasoning; we should *not* pay frontier-model prices for summarization. Instead, a small local model (Ollama + Llama-3.2-3B, or Qwen-2.5-3B, or similar ~3B-class) reads the session transcript and produces a sidecar update. Users who don't run Ollama can opt into a cheap API (e.g., Anthropic Haiku or OpenAI gpt-5-mini) for ~$0.002 per summary.

The sidecar schema has to serve two audiences: humans who might open the file and want to understand the codebase's history, and agents who want to retrieve context quickly. Markdown + YAML frontmatter splits the difference well. Structured fields (path, hash, dependencies, tags) go in YAML where they're machine-parseable; prose (reasoning, history, notes) lives in the Markdown body.

Sidecars live at `.ide/meta/<relative-path>.md` — mirroring the source tree under a single `.ide/meta/` root. This keeps the source tree clean (no `.meta` files polluting it) while keeping a one-to-one mapping trivial to reason about. The `.ide/` directory is hidden by default in the file tree but toggleable.

A separate concept — simple enough to include in this mission — is a project-level TODO list at `.ide/tasks.md`. Plain markdown checkboxes. Agents can read and write it through a dedicated tool. Humans can read and edit it. It's a small feature but it's the cheapest-possible implementation of something agents benefit from hugely: a durable list of what needs doing.

## Scope

**In scope:**
- New `editor-metadata` crate.
- Sidecar file format: Markdown + YAML frontmatter.
- Sidecar schema (frontmatter fields: source_path, source_hash, generated_by_model, generated_at, last_updated, dependencies, references, tags, summary).
- Sidecar body structure (sections: Summary, Reasoning, History, Dependencies, References, Notes).
- `MetadataStore` that manages sidecars, hashing, read/write.
- `Summarizer` trait with two implementations: `OllamaSummarizer` (local), `ApiSummarizer` (remote cheap API).
- Session capture: a `SessionLog` accumulates every ChatEvent + tool call + committed edit during an agent turn.
- Automatic sidecar update: after a session commits, the summarizer runs on affected files and updates their sidecars.
- Tools added to M20's registry: `read_metadata`, `write_metadata_note`, `add_task`, `list_tasks`, `complete_task`.
- Sidebar visibility toggle for `.ide/meta/` in settings.

**Out of scope:**
- Embedding / vector indexing of sidecars (M22).
- Automatic backfill: generating sidecars for files the agent never touched. V4+ can do a batch "document the whole codebase" pass.
- Rich editing UI for sidecars (users can just open them as regular Markdown files).
- Sidecar versioning beyond what git already provides.
- Cross-repo / multi-workspace metadata sharing (V4+).
- Access control on sidecars (they're checked into git like any other project file).

## North Star

An agent edits `src/auth.rs` to add a rate limiter. The session log captures: the user's prompt, the agent's plan, the four tool calls (read_file, grep, edit_lines, run_shell for tests), the final diff. When M23's chat panel commits the approved changes, a summarization job fires. The local summarizer reads the session log and produces:

```markdown
---
source_path: src/auth.rs
source_hash: a3f81...
generated_by_model: llama-3.2-3b-instruct
generated_at: 2026-05-14T16:20:00Z
last_updated: 2026-05-14T16:20:00Z
dependencies:
  - src/lib.rs
  - src/errors.rs
  - Cargo.toml (crates: tower, governor)
references:
  - https://docs.rs/governor/
tags: [auth, rate-limiting, security]
summary: Authentication handlers with a per-IP rate limiter backed by the governor crate.
---

# src/auth.rs

## Summary
Handles login, logout, and session validation. Rate-limited per IP to prevent
brute-force attacks. Uses the `governor` crate's token-bucket implementation.

## Reasoning
The rate limiter was added on 2026-05-14 in response to an observed surge in
failed login attempts against the staging environment. We considered three
approaches: (1) counting in Redis (rejected: extra dependency), (2) in-process
with `governor` (chosen: simple, fast, sufficient for one instance), (3)
reverse-proxy-level via Caddy (rejected: moves policy out of the app).

## History
- 2026-05-14 — Added per-IP rate limiter (5 req/min per IP, burst 10). Session #7b3.
- 2026-05-02 — Refactored session token generation to use UUIDv7. Session #4a1.
- 2026-04-21 — Initial implementation. Session #1c9.

## Dependencies
...
```

Now another agent, asked a week later about `src/auth.rs`, reads this sidecar in ~300 tokens rather than the 800-token source file plus whatever exploration it would have done to understand context. Reasoning accumulates. Tokens converted to durable project knowledge.

## Standing Orders Reminder

- Summarizer failures must never block a commit. Log, move on.
- Sidecar format is Markdown + YAML frontmatter. Resist the urge to invent a bespoke format.
- History is append-only. Never rewrite past entries; if a summary needs correcting, append a correction.
- The user can always delete a sidecar (it's just a file). Never treat sidecars as the authoritative source — they're derived metadata.
