# State of Project — 2026-04-20

**Context:** An audit of the current codebase found the project is ~38% done by mission count (M00-M10 shipped as `0.2.0-v2`), with M11-M13 partial and M14-M24 not yet started. The audit rated the codebase 8.5/10 — strong foundation, no structural rewrites needed, no critical security issues, a handful of specific bugs and four concrete gaps that must close before V3 can properly start.

This doc is the bridge between the audit findings and the new missions (M25-M28) that extend the existing plan. Read this first; then M25 is the next mission to execute.

## What the Audit Confirmed Is Working

- Text engine (M02), file I/O (M06), GPU rendering (M03/M04), input pipeline (M05), observability (M07), and all V2 polish (M09/M10) are shipped and the code quality is production-grade.
- Measured baselines meet or beat targets: p50 input-to-pixel ~2 ms, p95 ~5 ms, startup ~45 MB memory, 100 MB file loads in ~450 ms off the main thread.
- Architecture is clean — crate separation is correct, no circular deps, `#![forbid(unsafe_code)]` in application code, `cargo audit` clean, CI green on three OSes.
- No critical security or data-loss bugs. Atomic saves work correctly. Line-ending preservation round-trips correctly.

## What the Audit Confirmed Is Broken or Incomplete

Concrete fixes needed, ordered by severity:

1. **`EditorCommand::ToggleFullscreen` is defined in the enum but has no match arm** in `crates/editor-app/src/main.rs` around line 1032. (The audit's second pass reports the build currently passes, so this may have been resolved or may be a stub only caught by a narrow code path — verify by building with `-D warnings` and running the full test suite.)
2. **Workspace file-watcher events are discarded.** `poll_workspace_fs` in `main.rs:949` drains events but only `debug!()`-logs them. External file changes are invisible to the app. This is a direct M13 completion item.
3. **External-file-change save is silent.** If a file is modified externally while open, the editor flags it but does not block save — the user's version silently overwrites the external one. Data-loss risk, easy fix with a confirm-overwrite prompt.
4. **Layout cache is keyed incorrectly.** Keyed by `(line_index, version)` when it should be `(line_index, version, wrap_width)`. Pending M12 work.
5. **M11 installers not built.** Unsigned `.exe` on Releases; no MSI, no `.dmg`/notarization, no `.deb`/AppImage. Cannot ship to normal users today.
6. **M08 acceptance tests exist but are `#[ignore]`'d and never run in CI.** Perf regressions can land silently.

## Amendments to Existing Missions

The existing M12-M24 docs are still valid but the following adjustments apply when you execute them:

### M12 (Window, Resize, DPI)
- The layout-cache fix from the audit (item 4 above) is **already covered** by M12 TODO item 3 in the existing doc. M25 includes this work; you can treat it as done when M25 ships.

### M13 (Workspace & Multi-Buffer)
- The audit's workspace-watcher finding (item 2 above) is the completion of M13 TODO item 11. M25 also folds this in.

### M18 (Git Integration Baseline)
- The original M18 doc kept git strictly read-only (no stage/commit from the editor). That is still the right default for V3's editor UI, but M20's tool API *will* add `git_status`, `git_log`, `git_show`, `git_diff` tools so the AI can read git state programmatically. Write operations (stage, commit, branch-switch) remain out-of-scope for V3 UI and V3 AI — if the user wants to commit, they use the terminal (which, as of M26, is right there inline). M26's terminal makes this division cleaner: all git writes happen through the PTY, visible and auditable.

### M19 (AI Provider Abstraction)
- Ship order inside M19: **OpenAI first**, then Anthropic, then Gemini, then Ollama, then CustomHttp. Brian's guidance is "for now just to get things up and running and testing, we can use OpenAI." OpenAI's Chat Completions API is the oldest and most stable; its semantics are what most open-source local servers follow; starting there de-risks the abstraction. Pick `async-openai` as the client (mature, maintained, complete). Once OpenAI works end-to-end through M23's chat panel, add the other providers in the same interfaces.
- Tool-use note: OpenAI calls it `tools` + `tool_calls` in the response; Anthropic calls it `tool_use` content blocks. The provider trait already abstracts this; the OpenAI impl just maps OpenAI's shape into our uniform `ChatEvent::ToolCall`.

### M20 (Agent Tool-Use API)
- Add two new built-in tools on top of the original 12:
  - `run_terminal(command, timeout_seconds?)` — runs a command in the integrated terminal from M26 (not a fresh process). Output streams into the terminal *visibly* and is captured for the tool result. This is strictly better than the original `run_shell` tool for agent UX: the user *sees* what the agent ran and can intervene.
  - `read_terminal_output(lines_back)` — reads recent terminal scrollback so the agent can inspect previous command output without re-running anything.
- The original `run_shell` (direct `Command::spawn`) is kept as a fallback for non-interactive work but `run_terminal` becomes the preferred tool.

### M23 (AI Chat Panel)
- The panel's composer (the text input for user prompts) is what Brian described in the latest guidance. The original M23 doc already covers this; M28's settings panel is what makes it actually usable with real API keys.
- Reading M27's skills: the chat panel loads all installed skills' `name` + `description` into the system prompt at turn start (progressive disclosure — model sees what skills exist but not their content). When the model decides it needs a skill, it calls a `load_skill(name)` tool that returns the SKILL.md body.
