# V3 Vision: The AI-Native IDE

**Status:** Planning. Preceded by the V2 release (`0.2.x`). Implemented by missions M12 through M24.
**Scope horizon:** Mid-2026 through end-2026.
**Prerequisite:** `0.2.0-v2` (or `0.2.1` with installers from M11) must be shipped.

---

## The Horizon We're Walking Toward

By the end of 2026, the people writing most of the code on any given production project will not be humans. They will be coding agents — API calls to foundation models at Anthropic, OpenAI, Google, and others, plus locally-hosted open models running in Ollama or llama.cpp. Humans will still be in the loop: planning, reviewing, approving, debugging the non-obvious. But the keystrokes will increasingly come from something that is not a hand on a keyboard.

Today's editors were not designed for this. VS Code was designed in 2015 for humans typing code and using autocomplete. Cursor and Windsurf layered AI on top of VS Code's bones — they are very good, but they are retrofits. The opportunity is to build the first editor whose architecture treats "the agent" and "the person" as equally first-class users from the beginning — where every affordance that helps a person (file navigation, search, diff viewing, undo) has a symmetric, programmatic counterpart that an LLM can invoke through structured tool calls, where every action an agent takes flows through the same transaction system as a user's edit (so undo always works, so diffs always render, so the audit trail is always clean), and where the context an agent needs to do its job well is continuously maintained by the editor itself rather than reconstructed from scratch on every prompt.

That is the target. V3 is not a finished version of that target. V3 is the *foundational scaffolding* for it. It is the first version where:

- A person and an agent can both edit the same buffer through the same undo-aware transaction system.
- An agent can navigate the project's file system with precision tools, not free-form `cat` commands.
- Syntax highlighting and diff rendering make agent edits reviewable by humans at a glance.
- A vector index over the project makes "what's relevant to this prompt?" a solvable question rather than an expensive token gamble.
- A novel *metadata sidecar system* captures the reasoning behind each file — not just what the code does, but why it exists, what was considered, and what was decided — so that the expensive tokens spent by premium models are not lost at the end of every conversation but accumulated into long-lived project knowledge.

That last point is the one that makes this *distinct* rather than "a better Cursor." Today, every time a developer closes a Cursor conversation, tens of thousands of tokens of chain-of-thought, web search results, and design reasoning evaporate. All that survives is the final code diff. V3 bets that a disciplined sidecar-metadata system — hidden by default in the UI, machine-generated by a lightweight summarizer, vector-indexed for retrieval — turns those lost tokens into a compounding asset. A project that has been edited by agents for six months is a project where the agents now *know* why every file exists, which makes every subsequent task cheaper, faster, and more correct.

## What V3 Is Not

V3 is explicitly **not**:

- A finished AI-coding product. It is the infrastructure that makes a finished AI-coding product possible.
- A replacement for Cursor, Windsurf, or Claude Code. It is the first version where our editor enters the same category.
- Multi-user / collaborative editing. That is V4+ territory.
- Autonomous agents working unattended overnight. The V3 chat panel is a turn-based interaction; headless agent orchestration is V4+.
- Plugin ecosystem. Everything in V3 is first-party.
- A paid service. V3 ships as a local-first binary; cloud features are explicit opt-ins that call out to user-provided API keys.

## How V3 Preserves What V2 Earned

Every performance and correctness property that V2 validated in M08's acceptance pass — sub-5 ms input-to-pixel latency, stable memory over long sessions, atomic saves, fast open on 100 MB files — must stay green. Every mission in M12 through M24 ends with a check against the `m08-mvp` baseline. If a feature regresses the MVP, the feature is wrong until the regression is fixed. We add capability on top of the foundation; we do not degrade the foundation.

## The Missions, At a Glance

V3 is implemented by thirteen missions, working from foundational polish outward to the integrated AI experience:

- **M12** — Window, Resize, DPI: make the bones feel instant.
- **M13** — Workspace Model: project root, file watcher, ignore rules, multi-buffer backend.
- **M14** — File Tree, Tab Strip, Quick Open: the visible UI for M13.
- **M15** — Syntax Highlighting via Tree-sitter.
- **M16** — Find & Replace: in-file and project-wide.
- **M17** — Diff Engine & Inline Renderer: red/green hunks for file comparisons and, later, agent-edit previews.
- **M18** — Git Integration Baseline: status, branch, diff-vs-HEAD.
- **M19** — AI Provider Abstraction: Anthropic, OpenAI, Gemini, Ollama, custom endpoints.
- **M20** — Agent Tool-Use API: the structured surface LLMs call to read and edit the project.
- **M21** — Metadata Sidecar System: the novel per-file reasoning capture.
- **M22** — Local Vector Index: semantic retrieval over metadata + code.
- **M23** — AI Chat Panel & Edit-Approval Flow: the minimum viable integrated experience.
- **M24** — V3 Acceptance & Release: tagged `0.3.0-v3` with installers.

Each mission keeps the same shape as M00 through M11: narrow scope, ~15-18 granular TODOs, performance preservation required, quality gates, acceptance criteria, a tag on completion.

## The Mental Model

Think of V3 as four concentric rings, built outward:

1. **Ring 1 (M12-M14) — Polished substrate.** Window sizing is instant, DPI changes are clean, the editor holds multiple buffers, a project is an addressable thing with a tree view. The bones of a real IDE, not a text-file viewer.

2. **Ring 2 (M15-M18) — Developer affordances.** Syntax highlighting, search, diff, git status. The features that every editor ships and whose absence tells you immediately that a tool is unserious. We ship minimal versions that work correctly and leave room for depth later.

3. **Ring 3 (M19-M22) — Agent substrate.** Provider abstraction so any model works. Tool API so agents can act on the project safely. Metadata capture so reasoning accumulates. Vector index so context retrieval is cheap. This ring has no user-visible UI beyond "settings for providers and keys" — it is the agent-facing half of the editor.

4. **Ring 4 (M23) — Integration.** The chat panel. The first place where Rings 1-3 become a coherent experience for a person. Edits proposed by agents appear as diffs. Approval is per-hunk. Every operation flows through the same undo. The panel is deliberately minimal — a chat log, a stream of tokens, a list of tool calls, a diff viewer, accept/reject buttons. The polish of that UI is V4 territory; the correctness of the loop is V3.

5. **Shipping (M24) — The second release.** Acceptance tests, installers, tag `0.3.0-v3`. The first publicly-installable version with AI capabilities.

## On Not Overbuilding

Brian's direct guidance on this set: keep the basics, keep it simple, focus on the bones, don't bloat the UI, think foundations. Every mission carries an "Out of scope" section that is longer and more aggressive than the "In scope" section. When in doubt, the smaller choice is the right choice. The integrations we are building will support a hundred future features; what matters now is that the integrations are correct, fast, and composable.

## What Gets Deferred to V4+

- Autocomplete / inline AI suggestions as you type (requires tight coupling between input and inference that V3 doesn't ship).
- Multi-agent / background agents (V3 is turn-based).
- Plugin APIs / third-party extensions (V3 is closed).
- Collaborative editing (V3 is single-user).
- Debugger integration (V3 has no debug adapter).
- LSP client (V3 uses Tree-sitter for highlights only; real language intelligence is LSP, which is V4+).
- Custom themes beyond dark/light toggle (V3 ships a fixed dark theme plus a light variant).
- Remote editing / SSH / dev containers (V3 is local-first only).

Each of the above has a legitimate case and will get its own mission set when the time comes. None belong in V3.

## The Tests That Define V3 Success

By the end of M24:

1. A user opens a folder. The file tree appears. A file clicks open in under 50 ms. Syntax highlights render. The file edits with zero perceptible latency. All the V1/V2 acceptance numbers stay green.

2. A user types `Ctrl+P`, fuzzy-matches a filename, hits Enter, the file opens.

3. A user types `Ctrl+Shift+F`, types a query, gets a list of project-wide matches, clicks one, the corresponding file opens at the matching line.

4. A user opens the chat panel, types "add a rate limit to the `/api/login` handler," and an agent proposes a diff touching two files. The user reviews each hunk, accepts one, rejects one, and the accepted edit goes into the buffer with a single undo step.

5. Behind the scenes, the agent's reasoning (chain of thought, tools called, files read) is summarized by a local small model and written to sidecar metadata files next to the edited source. The next time an agent is asked about those files, it retrieves the accumulated metadata rather than reading the raw code from scratch.

6. A `0.3.0-v3` installer exists for Windows, macOS, and Linux. It installs cleanly on a VM. The first-launch experience walks the user through setting an API key (or pointing at a local Ollama endpoint). No data leaves the machine except through the user's configured provider.

If those six are true, V3 shipped. If any are false or sloppy, V3 is not done.
